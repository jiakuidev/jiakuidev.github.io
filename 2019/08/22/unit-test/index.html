单元测试规范

是指对软件中的最小可测试单元进行检查和验证

1. 单元测试目标

  1. 提升软件质量
  2. 促进代码优化
  3. 提升研发效率
  4. 增加从构自信

2. 单元测试的基本原则

宏观角度单元测试要复合AIR原则；
微观角度单元测试的代码层面要符合BCDE原则

1. AIR原则具体包括：

  • A: Automatic (自动化)
  • I: Independent (独立性)
  • R: Repeatable (可重复)

2. BCDE原则：

  • B: Border，边界值测试，包括循环边界、特殊取值、特殊时间点，数据顺序。
  • C: Correct，正确的输入，并得到预期的结果。
  • D: Design，与设计文档相结合，来编写单元测试。
  • E: Error，单元测试的目的是证明程序有错，而不是证明程序无错。为了发现代码中潜在的错误，我们需奥在编写测试用例时有一些强制的错误输入（如非法数据、异常流程、非业务允许输入等）来得到预期的错误结果

3. 需要Mock的因素

由于单元测试只是系统集成测试前的小模块测试，有些因素往往不具备的，因此需要进行Mock，例如：

  1. 功能因素：比如被测试方法内部调用的功能不可用。
  2. 时间因素：比如双十一还没有到来，与此时间相关的功能点。
  3. 环境因素：政策环境，如支付宝政策累新功能；多端环境，如PC、手机等。
  4. 数据因素：线下因素数据样本过小，难以覆盖线上真实场景。
  5. 其他因素：为了简化测试编写，开发者可以将一些复杂的依赖采用Mock的方式实现。

3. 单元测试的覆盖率

单元测试是一种白盒测试，测试者依据程序的内部结构来实现测试代码。单元测试覆盖率是指业务代码被单元测试用例测试的比例和程度，它是衡量单元测试好坏的一个很重要的指标，各类覆盖率指标从粗到细，从弱到强排列如下：

1. 粗力度覆盖

粗力度覆盖包括类覆盖和方法覆盖两种

  1. 类覆盖
    类覆盖是指只要有方法或者变量被测试用例调用到，那么就说明这个类被测试覆盖了。
  2. 方法覆盖
    只要测试执行过程中，某个方法被调用了则无论执行了改方法的多少行代码，都可以认为该方法被覆盖了。

2. 细力度的覆盖

细力度的覆盖包括一下几种

  1. 行覆盖（Line Coverage）
    行覆盖又称为语句覆盖，用来度量可执行的语句是否被执行到。

  1. 分支覆盖（Branch Coverage）
    分支覆盖也称为判定覆盖，用来度量每一个判定分支是否都被执行到。

  2. 条件判定覆盖（Condition Decision Coverge）
    条件判定覆盖要求设计足够的测试用例，能够让判定中每个条件的所有可能情况至少被执行一次，同时每个判定本身的所有可能结果也至少被执行一次。

  3. 条件组合覆盖（Multiple Condition Coverage）
    条件组合覆盖是指判定中所有条件的各种组合情况都出现至少一次。

    对于一个包含了n个条件的判定，至少需要2^n个测试用例才可以。虽然这种覆盖足够严谨，但会给测试用例增加了指数级的工作量。

  4. 路径覆盖（Path Coverage）
    路径覆盖要求能够测试到程序中所有可能的路径。
